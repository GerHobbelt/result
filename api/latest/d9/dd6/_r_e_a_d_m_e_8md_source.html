10 # Expected
11 
12 **Expected** is a simple and light-weight error-handling mechanism that offers
13 a non-throwing alternative to conventional exception handling.
14 
15 ![Expect the unexpected](doc/src/images/banner.png)
16 
17 ## Features
18 
19 * [x] Offers a coherent, light-weight alternative to exceptions
20 * [x] Compatible with <kbd>C++11</kbd> (with more features in <kbd>C++14</kbd> and <kbd>C++17</kbd>)
21 * [x] Single-header, **header-only** solution -- easily drops into any project
22 * [x] Support for value-type, reference-type, and `void`-type values in `expected`
23 * [x] Monadic composition functions like `map`, `flat_map`, and `map_error` for
24  easy functional use
25 * [x] [Comprehensively unit tested](test/src/expected.test.cpp) for both static
26  behavior and runtime validation
27 
28 For more details and examples on what is available in **Expected**, please
29 check out the [tutorial](doc/tutorial.md) section.

48 ## Background
49 
50 Error cases in C++ are often difficult to discern from the API. Any function
51 not marked `noexcept` can be assumed to throw an exception, but the exact _type_
52 of exception, and if it even derives from `std::exception`, is ambiguous.
53 Nothing in the language forces which exceptions may propagate from an API, which
54 can make dealing with such APIs complicated.
55 
56 Often it is more desirable to achieve `noexcept` functions where possible, since
57 this allows for better optimizations in containers (e.g. optimal moves/swaps)
58 and less cognitive load on consumers.
59 
60 Having an `expected<T, E>` type on your API not only semantically encodes that
61 a function is _able to_ fail, it also indicates to the caller _how_ the function
62 may fail, and what discrete, testable conditions may cause it to fail -- which
63 is what this library intends to solve.
64 
65 As a simple example, compare these two identical functions:
66 
67 ```cpp
68 // (1)
69 auto to_uint32(const std::string& x) -> std::uint32_t;
70 
71 // (2)
72 enum class parse_error { overflow=1, underflow=2, bad_input=3};
73 auto to_uint32(const std::string& x) noexcept -> expected<std::uint32_t,parse_error>;
74 ```
75 
76 In `(1)`, it is ambiguous _what_ (if anything) this function may throw on
77 failure, or how this error case may be accounted for.
78 
79 In `(2)`, on the other hand, it is explicit that `to_uint32` _cannot_ throw --
80 so there is no need for a `catch` handler. It's also clear that it may fail for
81 whatever reasons are in `parse_error`, which discretely enumerates any possible
82 case for failure.
83 
84 ### Deviations from `std::expected` proposals
85 
86 Although this library is heavily based on the `std::expected` proposal, it's
87 important to be aware that this handles a few things differently from what is
88 described in `P0323`. These deviations are outline below
89 
90 1. This does not implement `emplace` functions, since this functionality is
91  seen as orthogonal to the goals of `expected`, which should not be to
92  arbitrarily be an `either<T,E>` type
93 
94 2. `expected` has been given support for reference `T` types, so that this may
95  behave as a consistent vocabulary type for error-handling
96 
97 3. `unexpected` has been given support for reference `E` types, in order to
98  avoid expensive construction of objects only used for comparisons
99 
100 4. Assignment operators are only enabled if the corresponding constructors are
101  marked `noexcept`. This deviates from `std::expected`'s proposal of
102  introducing an intermediate object to hold the type during assignment.
103 
104 5. Rather than allowing direct referential access to the underlying error,
105  `expected::error()` _always_ returns a value that acts as the result's
106  status. The defaut-constructed state of `E` is always considered the "good"
107  state (e.g. `std::error_code{}`, `std::errc{}`, etc). This reduces the
108  number of cases where this API may throw an exception to just `value()`
109 
110 6. Rather than using `unexpect_t` to denote in-place construction of errors,
111  this library uses `in_place_error_t`. This is done to prevent having 3 types
112  that all sound similar (`expected`, `unexpected`, `unexpect_t`)
