15 **Result** is a modern, simple, and light-weight error-handling alternative to exceptions.

17 ## Teaser
18 
19 ```cpp
20 template <typename To, typename From>
21 auto try_narrow(const From& from) noexcept -> cpp::result<To,narrow_error>
22 {
23  const auto to = static_cast<To>(from);
24 
25  if ((to < To{}) != (from < From{})) {
26  return cpp::fail(narrow_error::sign_change);
27  }
28  if (static_cast<From>(to) != from) {
29  return cpp::fail(narrow_error::loss_of_data);
30  }
31  return to;
32 }
33 ```

37 ## Features
38 
39 * [x] Offers a coherent, light-weight alternative to exceptions
40 * [x] Compatible with <kbd>C++11</kbd> (with more features in <kbd>C++14</kbd> and <kbd>C++17</kbd>)
41 * [x] Single-header, **header-only** solution -- easily drops into any project
42 * [x] Zero overhead abstractions -- don't pay for what you don't use.
43 * [x] No dependencies
44 * [x] Support for value-type, reference-type, and `void`-type values in `result`
45 * [x] Monadic composition functions like `map`, `flat_map`, and `map_error` for
46  easy functional use
47 * [x] [Comprehensively unit tested](https://coveralls.io/github/bitwizeshift/result?branch=master) for both static
48  behavior and runtime validation
49 * [x] [Incurs minimal cost when optimized](https://godbolt.org/z/M69T4v), especially for trivial types
50

74 ## Background
75 
76 Error cases in C++ are often difficult to discern from the API. Any function
77 not marked `noexcept` can be assumed to throw an exception, but the exact _type_
78 of exception, and if it even derives from `std::exception`, is ambiguous.
79 Nothing in the language forces which exceptions may propagate from an API, which
80 can make dealing with such APIs complicated.
81 
82 Often it is more desirable to achieve `noexcept` functions where possible, since
83 this allows for better optimizations in containers (e.g. optimal moves/swaps)
84 and less cognitive load on consumers.
85 
86 Having a `result<T, E>` type on your API not only semantically encodes that
87 a function is _able to_ fail, it also indicates to the caller _how_ the function
88 may fail, and what discrete, testable conditions may cause it to fail -- which
89 is what this library intends to solve.
90 
91 As a simple example, compare these two identical functions:
92 
93 ```cpp
94 // (1)
95 auto to_uint32(const std::string& x) -> std::uint32_t;
96 
97 // (2)
98 enum class parse_error { overflow=1, underflow=2, bad_input=3};
99 auto to_uint32(const std::string& x) noexcept -> result<std::uint32_t,parse_error>;
100 ```
101 
102 In `(1)`, it is ambiguous _what_ (if anything) this function may throw on
103 failure, or how this error case may be accounted for.
104 
105 In `(2)`, on the other hand, it is explicit that `to_uint32` _cannot_ throw --
106 so there is no need for a `catch` handler. It's also clear that it may fail for
107 whatever reasons are in `parse_error`, which discretely enumerates any possible
108 case for failure.

110 ## Optional Features
111 
112 Although not required or enabled by default, **Result** supports two optional
113 features that may be controlled through preprocessor symbols:
114 
115 1. Using a custom namespace, and
116 2. Disabling all exceptions
117 
118 ### Using a Custom Namespace
119 
120 The `namespace` that `result` is defined in is configurable. By default,
121 it is defined in `namespace cpp`; however this can be toggled by defining
122 the preprocessor symbol `RESULT_NAMESPACE` to be the name of the desired
123 namespace.
124 
125 This could be done either through a `#define` preprocessor directive:
126 
127 ```cpp
128 #define RESULT_NAMESPACE example
129 #include <result.hpp>
130 
131 auto test() -> example::result<int,int>;
132 ```
133 
134 <kbd>[Try Online](https://godbolt.org/z/a4GccT)</kbd>
135 
136 Or it could also be defined using the compile-time definition with `-D`, such
137 as:
138 
139 `g++ -std=c++11 -DRESULT_NAMESPACE=example test.cpp`
140 
141 ```cpp
142 #include <result.hpp>
143 
144 auto test() -> example::result<int,int>;
145 ```
146 
147 <kbd>[Try Online](https://godbolt.org/z/5xTsdj)</kbd>
148 
149 ### Disabling Exceptions
150 
151 Since `result` serves to act as an orthogonal/alternative error-handling
152 mechanism to exceptions, it may be desirable to not have _any_ exceptions at
153 all. IF the compiler has been configured to disable exception entirely, simply
154 having a path that even encounters a `throw` -- even if never reached in
155 practice may trigger compile errors.
156 
157 To account for this possibility, **Result** may have exceptions removed by
158 defining the preprocessor symbol `RESULT_DISABLE_EXCEPTIONS`.
159 
160 Note that if this is done, contract-violations will now behave differently:
161 
162 * Contract violations will call `std::abort`, causing immediate termination
163  (and often, core-dumps for diagnostic purposes)
164 * Contract violations will print directly to `stderr` to allow context for the
165  termination
166 * Since exceptions are disabled, there is no way to perform a proper stack
167  unwinding -- so destructors will _not be run_. There is simply no way to
168  allow for proper RAII cleanup without exceptions in this case.
169 
170 <kbd>[Try Online](https://godbolt.org/z/bjbqaG)</kbd>

172 ## Building the Unit Tests
173 
174 Building the unit tests are not necessary to use this project. However, if
175 you are interested in running these yourself, you will require
176 the following installed:
177 
178 * [CMake](https://cmake.org): Used for configuring/building the project
179 * [Catch2](https://github.com/catchorg/Catch2): the unit-test library
180 
181 Additionally, you will need to toggle the `RESULT_COMPILE_UNIT_TESTS` option
182 during cmake configuration to ensure that unit tests configure and build.
183 
184 The easiest way to install Catch2 is using the [`conan`](https://conan.io/index.html)
185 package manager and installing with `conan install <path to conanfile>` from your
186 build directory.
187 
188 A complete example of configuring, compiling, and running the tests:
189 
190 ```sh
191 # Make the build directory and enter it
192 mkdir build && cd build
193 # Install Catch with conan (optional)
194 conan install ..
195 # Configure the project
196 cmake .. -DRESULT_COMPILE_UNIT_TESTS=On
197 # Build everything
198 cmake --build .
199 # run the tests
200 cmake --build . --target test
201 ```

203 ## Compiler Compatibility
204 
205 **Result** is compatible with any compiler capable of compiling valid
206 <kbd>C++11</kbd>. Specifically, this has been tested and is known to work
207 with:
208 
209 * GCC 5, 6, 7, 8, 9, 10
210 * Clang 3.5, 3.6, 3.7, 3.8, 3.9, 4, 5, 6, 7, 8, 9, 10, 11
211 * Apple Clang (Xcode) 10.3, 11.2, 11.3, 12.3
212 * Visual Studio 2017<sup>[1]</sup>, 2019
213 
214 Latest patch level releases are assumed in the versions listed above.
215 
216 **Note:** Visual Studios 2015 is not currently supported due to an internal
217 compiler error experienced in the default constructor of `result`. Support for
218 this will be added at a later time.
219 
220 <sup>[1] Visual Studios 2017 is officially supported, though toolchain 14.16
221 has some issues properly compiling `map_error` due to insufficient support for
222 SFINAE.</sup>
