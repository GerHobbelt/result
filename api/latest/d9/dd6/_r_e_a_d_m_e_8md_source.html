15 **Result** is a modern, simple, and light-weight error-handling alternative to
16 exceptions with a rich feature-set.
17 
18 ## Features
19 
20 ✔️ Offers a coherent, light-weight alternative to exceptions \
21 ✔️ Compatible with <kbd>C++11</kbd> (with more features in <kbd>C++14</kbd> and <kbd>C++17</kbd>) \
22 ✔️ Single-header, **header-only** solution -- easily drops into any project \
23 ✔️ Zero overhead abstractions -- don't pay for what you don't use. \
24 ✔️ No dependencies \
25 ✔️ Support for value-type, reference-type, and `void`-type values in `result` \
26 ✔️ Monadic composition functions like `map`, `flat_map`, and `map_error` for
27  easy functional use \
28 ✔️ Optional support to disable all exceptions and rename the `cpp` `namespace` \
29 ✔️ [Comprehensively unit tested](https://coveralls.io/github/bitwizeshift/result?branch=master) for both static
30  behavior and runtime validation \
31 ✔️ [Incurs minimal cost when optimized](https://godbolt.org/z/TsonT1), especially for trivial types
32 
33 Check out the [tutorial](doc/tutorial.md) to see what other features **Result**
34 offers.
35 
36 If you're interested in how `cpp::result` deviates from `std::expected`
37 proposals, please see [this page](doc/deviations-from-proposal.md).

41 ```cpp
42 enum class narrow_error{ none, loss_of_data };
43 
44 template <typename To, typename From>
45 auto try_narrow(const From& from) noexcept -> cpp::result<To,narrow_error>
46 {
47  const auto to = static_cast<To>(from);
48 
49  if (static_cast<From>(to) != from) {
50  return cpp::fail(narrow_error::loss_of_data);
51  }
52 
53  return to;
54 }

61 } to_string;

87 ## Why `result`?
88 
89 Error cases in C++ are often difficult to discern from the API. Any function
90 not marked `noexcept` can be assumed to throw an exception, but the exact _type_
91 of exception, and if it even derives from `std::exception`, is ambiguous.
92 Nothing in the language forces which exceptions may propagate from an API, which
93 can make dealing with such APIs complicated.
94 
95 Often it is more desirable to achieve `noexcept` functions where possible, since
96 this allows for better optimizations in containers (e.g. optimal moves/swaps)
97 and less cognitive load on consumers.
98 
99 Having a `result<T, E>` type on your API not only semantically encodes that
100 a function is _able to_ fail, it also indicates to the caller _how_ the function
101 may fail, and what discrete, testable conditions may cause it to fail -- which
102 is what this library intends to solve.
103 
104 As a simple example, compare these two identical functions:
105 
106 ```cpp
107 // (1)
108 auto to_uint32(const std::string& x) -> std::uint32_t;
109 
110 // (2)
111 enum class parse_error { overflow=1, underflow=2, bad_input=3};
112 auto to_uint32(const std::string& x) noexcept -> result<std::uint32_t,parse_error>;
113 ```
114 
115 In `(1)`, it is ambiguous _what_ (if anything) this function may throw on
116 failure, or how this error case may be accounted for.
117 
118 In `(2)`, on the other hand, it is explicit that `to_uint32` _cannot_ throw --
119 so there is no need for a `catch` handler. It's also clear that it may fail for
120 whatever reasons are in `parse_error`, which discretely enumerates any possible
121 case for failure.

123 ## Compiler Compatibility
124 
125 **Result** is compatible with any compiler capable of compiling valid
126 <kbd>C++11</kbd>. Specifically, this has been tested and is known to work
127 with:
128 
129 * GCC 5, 6, 7, 8, 9, 10
130 * Clang 3.5, 3.6, 3.7, 3.8, 3.9, 4, 5, 6, 7, 8, 9, 10, 11
131 * Apple Clang (Xcode) 10.3, 11.2, 11.3, 12.3
132 * Visual Studio 2017<sup>[1]</sup>, 2019
133 
134 Latest patch level releases are assumed in the versions listed above.
135 
136 **Note:** Visual Studios 2015 is not currently supported due to an internal
137 compiler error experienced in the default constructor of `result`. Support for
138 this will be added at a later time.
139 
140 <sup>[1] Visual Studios 2017 is officially supported, though toolchain 14.16
141 has some issues properly compiling `map_error` due to insufficient support for
142 SFINAE.</sup>
